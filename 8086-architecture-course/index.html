Registers, Flags & Memory

The CPU's inner world — its scratchpad, mood ring, and vast storage vault

The CPU's scratchpad: 14 registers

Think of registers like the sticky notes on a chef's workstation — a tiny handful of labeled spots where the cook keeps the numbers they're actively working with right now. The 8086 has 14 of them:

General Purpose (Arithmetic)

AX Accumulator
BX Base
CX Counter
DX Data

Each is 16 bits wide. AX can be split into AH (high byte) and AL (low byte).

Index & Pointer

SI Source Index
DI Dest Index
SP Stack Pointer
BP Base Pointer

Used for pointing to locations in memory. SP tracks the top of the stack. IP (not shown) holds the next instruction address.

Segment Registers

CS Code Segment
DS Data Segment
SS Stack Segment
ES Extra Segment

These point to 64 KB "windows" into the 1 MB address space. CS:IP together point to the next instruction.

What registers look like in the emulator code

All CPU state lives in a single JavaScript object. Registers are just properties of that object — nothing magical:

CPU8086.js 

const getReg16 = (idx) =>
  e.reg[
    ["AX","CX","DX","BX",
     "SP","BP","SI","DI"][idx]
  ];
const setReg16 = (idx, val) =>
  e.reg[["AX","CX","DX","BX",
         "SP","BP","SI","DI"][idx]]
    = val & 0xFFFF;
PLAIN ENGLISH

Define a function that reads a 16-bit register by its numeric index...

Look up the register name in an ordered list (AX=0, CX=1, DX=2...) ...

...the 8086 hardware defines this exact order — it matches the CPU encoding.

Define a function that writes a value to a 16-bit register...

...same name lookup using the index number...

...but mask to 16 bits (0xFFFF) so it never stores more than 65535. This is how real hardware works.

Flags — the CPU's mood indicators

After every calculation, the CPU automatically updates a set of flags. Think of them like dashboard warning lights on a car — they tell you what just happened:

ZF

Zero Flag

Set to 1 when the result of an operation is zero. Powers instructions like JZ (jump if zero).

CF

Carry Flag

Set when an addition overflows or a subtraction underflows. Essential for multi-byte arithmetic and detecting errors.

SF

Sign Flag

Mirrors the top bit of the result. If SF=1, the result is negative (in signed arithmetic). Powers JL / JG comparisons.

DF

Direction Flag

Controls whether string operations move forward (DF=0) or backward (DF=1) in memory. Set with STD / cleared with CLD.

IF

Interrupt Flag

When IF=1, hardware interrupts (keyboard, timer) are allowed. When IF=0, the CPU ignores them — like "do not disturb" mode.

The 1 MB memory vault — accessed via zip codes

The 8086 has a 1 MB address space. But the registers are only 16-bit, which can only address 64 KB. How do you reach all 1 MB? With a clever trick called segmented addressing.

Think of memory like a city with ZIP codes. The segment register is the ZIP code (neighborhood). The offset register is the street address. Together they pinpoint any house in the city:

CPU8086.js 

// Total address space (1 MB)
const ADDR_SPACE = 1048576;

// Convert segment:offset to physical address
const calcPhys = (s, o) =>
  (((s & 0xFFFF) << 4)
  + (o & 0xFFFF))
  & (ADDR_SPACE - 1);
PLAIN ENGLISH

The entire memory is 1,048,576 bytes (1 MB) — defined as a constant.

A function that converts a segment + offset pair to a single physical address...

Take the segment value and shift it left by 4 bits (multiply by 16). This gives a 20-bit base address.

Add the offset value to it...

...and wrap around if it exceeds 1 MB (the & mask keeps it in range). Result: any byte in the 1 MB vault is reachable.

🗺️
The formula: physical = (segment × 16) + offset

Example: CS=0x0000, IP=0x0100 → physical address = (0 × 16) + 256 = 256. That's where your program's first instruction lives when you use ORG 100h. You can verify this in the Memory Viewer!

The 1 MB memory map — neighborhoods with reserved addresses

Not all of memory is equally free. Certain address ranges are reserved for hardware — like land zoned for government buildings:

0x00000–0x003FF
Interrupt Vector Table — 256 pointers to interrupt handlers (4 bytes each)
0x00400–0x004FF
BIOS Data Area (BDA) — keyboard buffer, cursor position, hardware status
0x07C00
Boot Sector Load Point — BIOS copies disk sector 0 here, then jumps to it
0xB8000–0xBBFFF
VGA Text VRAM — the 80×25 screen. Write here → characters appear on the monitor
0xFFFFF
Reset Vector — the very first address the CPU executes after power-on

How the emulator packs 8 flags into one 16-bit number

All flags are stored in a single 16-bit flags register. The emulator uses two functions — packFlags and unpackFlags — to convert between the object form and the raw hardware integer:

CPU8086.js 

const packFlags = (e) => {
  const f = e.flags; let r = 0;
  if (f.CF) r |= (1<<0);
  if (f.ZF) r |= (1<<6);
  if (f.SF) r |= (1<<7);
  if (f.IF) r |= (1<<9);
  if (f.DF) r |= (1<<10);
  if (f.OF) r |= (1<<11);
  return r;
};
PLAIN ENGLISH

A function to pack all flags into one 16-bit hardware number...

Start with result = 0 (all bits off)...

If Carry Flag is set, turn on bit 0 of the result (hardware position for CF).

If Zero Flag is set, turn on bit 6 (the 8086 hardware specification says ZF lives at bit 6).

Sign Flag → bit 7. Interrupt Flag → bit 9. These positions are fixed by Intel's specification.

Direction Flag → bit 10. Overflow Flag → bit 11.

Return the single packed number — this is the exact format real 8086 hardware uses in the PUSHF instruction.

How a simple instruction changes everything

Let's trace what happens when the CPU executes CMP AX, 0 — the instruction asks "is AX equal to zero?"

IP
Instruction Pointer (CS:IP)

Fetching next instruction from memory at 0x0100... it says CMP AX, 0

DE
Binary Decoder

I decoded byte 0x3D from memory. That means CMP AX, immediate. Fetching the 16-bit value to compare against...

AX
AX Register

My current value is 0x0000 (zero). The comparison is: 0 - 0 = 0.

FL
Flags Register

Result is zero! I'm setting ZF=1, SF=0, CF=0, OF=0. Now any JZ instruction will jump!

IP
Instruction Pointer

IP advances past CMP to the next instruction. Ready for whatever comes next — JZ, JNZ, or just the next statement.

Speaking Machine Language

How human-readable assembly becomes the raw bytes a CPU can eat

Assembly is a thin wrapper around machine code

Assembly language is the thinnest possible abstraction over raw binary. Instead of writing bytes like 0xB8 0x00 0x00, you write mov ax, 0 and let the assembler do the translation.

Think of it like Morse code — a human-readable shorthand that maps precisely to a binary signal:

Assembly Text
mov ax, 0xB800
mov es, ax
mov byte [es:di], 65
→
Machine Code Bytes
B8 00 B8
8E C0
26 88 05

The assembler works in two passes — like a book editor

The Assembler8086 class uses a classic two-pass technique. Imagine editing a book where chapter 3 references "see Chapter 7" — you can't know the page number until you've laid out the whole book first:

1

Pass 1 — Scout: Scan all lines, calculate instruction sizes, build a symbol table of every label and its address

→
2

Pass 2 — Encode: Go through every instruction again, now generating actual bytes using the symbol table to resolve labels

→
✓

Output: A Uint8Array of machine code bytes, ready to load into the CPU's memory

📚
Why two passes?

In one-pass assemblers, you can only jump to labels you've already seen. Two-pass lets you write JMP END_LABEL before END_LABEL is defined. Every serious assembler (NASM, GAS, MASM) uses this pattern.

Every instruction is a rule: operand types → bytes

The assembler stores a giant opcode table — a recipe book where each instruction mnemonic maps to rules for encoding it:

Assembler8086.js 

const OPCODE_TABLE = {
  'HLT': [{
    match: [],
    size: () => 1,
    encode: () => [0xF4]
  }],
  'INT': [{
    match: [OpType.NUMBER],
    size: (ops) =>
      (ops[0].value === 3) ? 1 : 2,
    encode: (ops) => {
      if (ops[0].value === 3)
        return [0xCC];
      return [0xCD, ops[0].value & 0xFF];
    }
  }]
};
PLAIN ENGLISH

A big lookup object: instruction names → encoding rules.

HLT (halt): takes no operands...

...always produces exactly 1 byte...

...and that byte is 0xF4 — the 8086 opcode for halt.

INT (software interrupt): takes one number as operand...

...INT 3 is special: it is 1 byte (0xCC). All other INTs are 2 bytes...

...INT 3 encodes as [0xCC] (breakpoint shorthand). All others: [0xCD, interrupt-number].

The ModRM byte — 8086's clever compression trick

Most 8086 instructions use a single extra byte called ModRM to specify their operands. It's like an area code — a short numeric code that tells the CPU exactly where to find each piece of data:

Mod (2 bits)
Reg (3 bits)
RM (3 bits)

00 = [mem]

01 = [mem+disp8]

10 = [mem+disp16]

11 = register

Destination register or opcode extension

000=AX, 001=CX, 010=DX, 011=BX, 100=SP, 101=BP, 110=SI, 111=DI

Source register or memory base

Same codes as Reg field — but when Mod=11, it picks a register. Otherwise picks a memory addressing mode.

Assembler8086.js 

const encodeModRM = (mod, reg, rm) =>
  ((mod & 0b11)   << 6)
  | ((reg & 0b111) << 3)
  | (rm  & 0b111);
PLAIN ENGLISH

Pack three values into one byte using bit shifting.

Put the Mod field (2 bits) into the top two bits of the byte (positions 7-6).

Put the Reg field (3 bits) into the middle (positions 5-3).

Put the RM field (3 bits) into the bottom (positions 2-0). One byte = three fields = all operand info.

The Heartbeat

Fetch → Decode → Execute: the three-step drum beat every CPU lives by

Every CPU in history runs the same three-step loop

From the 8086 to your MacBook's M-series chip, every CPU runs an infinite loop called the Fetch-Decode-Execute cycle. Think of it like a music box: a tiny pin plucks one note, then the cylinder rotates to the next pin, forever.

📥

FETCH

Read the next byte from memory at address CS:IP. Advance IP.

→
🔍

DECODE

Identify the instruction: is it MOV? ADD? JMP? Read any extra bytes (operands).

→
⚡

EXECUTE

Perform the operation: update registers, write memory, set flags. Loop back to FETCH.

FETCH: the first beat

At the start of every instruction, the CPU reads the next byte from memory using CS:IP as the address. IP automatically advances after each read:

CPU8086.js — executeBinaryStep 

const fetch8 = () => {
  const v = readMem8(
    e,
    calcPhys(e.reg.CS, e.reg.IP)
  );
  e.reg.IP =
    (e.reg.IP + 1) & 0xFFFF;
  return v;
};
const fetch16 = () => {
  const lo = fetch8();
  const hi = fetch8();
  return (hi << 8) | lo;
};
PLAIN ENGLISH

fetch8: reads one byte from memory...

...calculate the physical address from segment CS and offset IP...

...read the byte there...

...advance IP by 1 (wrapped to 16 bits so it wraps at 65535)...

...return the byte. IP now points to the byte after the one we just read.

fetch16: reads two bytes to form a 16-bit word...

...read the low byte first (x86 is little-endian)...

...then the high byte...

...combine them: high byte shifted left 8 bits + low byte = 16-bit value.

DECODE: the CPU's giant menu

After fetching the first byte (the opcode), the decoder looks it up in a massive switch statement — one case per possible first byte:

CPU8086.js — executeBinaryStep (simplified) 

let opcode = fetch8();

switch (opcode) {
  case 0xF4:
    // HLT — stop execution
    return false;

  case 0x90:
    // NOP — do nothing
    break;

  case 0xB8: // MOV AX, imm16
    e.reg.AX = fetch16();
    break;
  // ... 200+ more cases ...
}
PLAIN ENGLISH

Fetch the first byte — this is the opcode, the instruction ID number.

A big decision tree: what does this byte mean?

0xF4 is the halt instruction...

...return false to signal "stop running."

0x90 is NOP — No OPeration. Literally do nothing (used for timing padding).

Break to the next iteration of the run loop.

0xB8 is "MOV AX, immediate 16-bit value"...

...fetch the next 2 bytes from memory (the number to load) and put them in AX.

There are 200+ opcodes total. The switch handles all of them.

🔑
The opcode IS the instruction

Machine code is not magic — it is just numbers. 0xF4 always means HLT. 0xB8 always means MOV AX. The CPU's switch statement is its entire vocabulary. That switch statement is what people mean when they say "hardware instruction set."

The run loop: 20 instructions every animation frame

When you click "Run," the emulator uses requestAnimationFrame to run the CPU in bursts — executing about 20 instructions per frame, then handing control back to the browser to update the UI:

App.jsx — runLoop 

const runLoop = useCallback(() => {
  if (!runningRef.current) return;
  for (let i = 0; i < 20; i++) {
    const ok =
      executeStep(eng.current, ctx);
    if (!ok) {
      runningRef.current = false;
      break;
    }
  }
  forceRender();
  requestAnimationFrame(runLoop);
}, []);
PLAIN ENGLISH

The main run loop function — it schedules itself to run every animation frame (~60fps).

If the user clicked Stop, quit immediately.

Execute up to 20 CPU instructions in a tight burst...

...call executeStep() once per instruction. It returns true to continue or false to halt.

If the CPU halted (HLT instruction or error)...

...mark running=false and stop the loop.

After the burst: force the React UI to re-render (show updated registers, VGA screen, etc.)

Schedule the next burst 1/60th of a second later via requestAnimationFrame — keeps the UI responsive.

The Outside World

BIOS interrupts, VGA memory, keyboard, disk, and the boot sequence

The CPU is not an island

A CPU alone can't do much. It needs to talk to the outside world — the screen, keyboard, disk, speaker. It does this via two channels: memory-mapped I/O and port I/O. Think of it like a hotel: room-to-room calls go through the building wiring (memory-mapped), but calls outside go through the front desk (port I/O).

🖥️

VGA Screen

Write character bytes to 0xB8000 in memory and they appear on screen. No special instruction needed — it's just a memory write.

⌨️

Keyboard

Port 0x60 streams scan codes. The emulator also supports BIOS INT 16h for higher-level keyboard reading.

🔊

PC Speaker

Port 0x61 controls the speaker gate. Writing the frequency divisor to port 0x42 sets the pitch. Implemented via Web Audio API.

💿

Disk

Port 0x70 selects a sector. Port 0x71 transfers it to/from memory at DS:BX. Or use BIOS INT 13h for sector reads/writes.

BIOS interrupts — the original operating system API

Before modern operating systems, programs called the BIOS directly for hardware services. The mechanism is the INT instruction — a software interrupt. Like pressing a service button in an elevator:

INT 10h
Video Services
AH=00 → Set video mode / clear screen
AH=02 → Set cursor position (DH=row, DL=col)
AH=09 → Write character+attr at cursor (AL=char, BL=attr)
AH=0E → Write char in TTY mode (auto-scroll)
INT 16h
Keyboard Services
AH=00 → Wait for a keypress, return in AX
AH=01 → Check if key available (ZF=0 if yes)
AH=02 → Get shift/ctrl/alt key status byte
INT 13h
Disk Services
AH=02 → Read sectors to ES:BX buffer
AH=03 → Write sectors from ES:BX buffer
AH=08 → Get drive geometry (cylinders, heads, sectors)

Inside the BIOS video handler

When your assembly code calls INT 10h, control jumps to handleInt10 in CPU8086.js:

CPU8086.js — handleInt10 (partial) 

const handleInt10 = (e) => {
  const ah =
    (e.reg.AX >> 8) & 0xFF;
  const al =
    e.reg.AX & 0xFF;
  if (ah === 0x0E) {
    const idx = VGA_BASE
      + (e.cursorY * 80
      + e.cursorX) * 2;
    e.mem[idx]     = al;
    e.mem[idx + 1] = 0x07;
    e.cursorX++;
  }
};
PLAIN ENGLISH

The BIOS video interrupt handler — called whenever assembly code runs INT 10h.

Read the high byte of AX (the "function selector")...

...AH tells us which video service was requested.

Read the low byte of AX (the character to print).

AH=0x0E means "TTY write" — print a character at the cursor position.

Calculate the VGA memory address: base 0xB8000 + (row × 80 + column) × 2 bytes per cell.

Write the character byte to that address. Now it shows on screen!

Write 0x07 (white on black) as the color attribute for this cell.

Move the cursor one column to the right — ready for the next character.

The boot sequence — from power-on to running code

When you click the Boot button in the emulator, it replays the exact sequence a real PC does at startup. Understanding this is knowing the blueprint of every operating system ever written:

CPU
CPU (after power-on)

I just turned on. My IP is set to 0xFFF0 — the reset vector. I start executing whatever code is there.

BIOS
BIOS (at reset vector)

I load disk sector 0 (the first 512 bytes of the virtual disk) into memory at address 0x7C00...

BIOS
BIOS

I check the last two bytes of sector 0: are they 0x55 and 0xAA? That is the boot signature — it means "this disk is bootable." If not, I refuse to boot.

CPU
CPU

Boot signature found! BIOS sets CS=0x0000 and IP=0x7C00. I jump to 0x7C00 and start executing the boot sector code. This could be a bootloader, or a tiny OS, or a Hello World.

DISK
Virtual Disk

I am a 128 KB array. Sectors 0-255 each hold 512 bytes. You can write bootable code to me using the Disk Viewer, then click Boot to run it!
