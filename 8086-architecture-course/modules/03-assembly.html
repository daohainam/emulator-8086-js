Speaking Machine Language

How human-readable assembly becomes the raw bytes a CPU can eat

Assembly is a thin wrapper around machine code

Assembly language is the thinnest possible abstraction over raw binary. Instead of writing bytes like 0xB8 0x00 0x00, you write mov ax, 0 and let the assembler do the translation.

Think of it like Morse code — a human-readable shorthand that maps precisely to a binary signal:

Assembly Text
mov ax, 0xB800
mov es, ax
mov byte [es:di], 65
→
Machine Code Bytes
B8 00 B8
8E C0
26 88 05

The assembler works in two passes — like a book editor

The Assembler8086 class uses a classic two-pass technique. Imagine editing a book where chapter 3 references "see Chapter 7" — you can't know the page number until you've laid out the whole book first:

1

Pass 1 — Scout: Scan all lines, calculate instruction sizes, build a symbol table of every label and its address

→
2

Pass 2 — Encode: Go through every instruction again, now generating actual bytes using the symbol table to resolve labels

→
✓

Output: A Uint8Array of machine code bytes, ready to load into the CPU's memory

📚
Why two passes?

In one-pass assemblers, you can only jump to labels you've already seen. Two-pass lets you write JMP END_LABEL before END_LABEL is defined. Every serious assembler (NASM, GAS, MASM) uses this pattern.

Every instruction is a rule: operand types → bytes

The assembler stores a giant opcode table — a recipe book where each instruction mnemonic maps to rules for encoding it:

Assembler8086.js 

const OPCODE_TABLE = {
  'HLT': [{
    match: [],
    size: () => 1,
    encode: () => [0xF4]
  }],
  'INT': [{
    match: [OpType.NUMBER],
    size: (ops) =>
      (ops[0].value === 3) ? 1 : 2,
    encode: (ops) => {
      if (ops[0].value === 3)
        return [0xCC];
      return [0xCD, ops[0].value & 0xFF];
    }
  }]
};
PLAIN ENGLISH

A big lookup object: instruction names → encoding rules.

HLT (halt): takes no operands...

...always produces exactly 1 byte...

...and that byte is 0xF4 — the 8086 opcode for halt.

INT (software interrupt): takes one number as operand...

...INT 3 is special: it is 1 byte (0xCC). All other INTs are 2 bytes...

...INT 3 encodes as [0xCC] (breakpoint shorthand). All others: [0xCD, interrupt-number].

The ModRM byte — 8086's clever compression trick

Most 8086 instructions use a single extra byte called ModRM to specify their operands. It's like an area code — a short numeric code that tells the CPU exactly where to find each piece of data:

Mod (2 bits)
Reg (3 bits)
RM (3 bits)

00 = [mem]

01 = [mem+disp8]

10 = [mem+disp16]

11 = register

Destination register or opcode extension

000=AX, 001=CX, 010=DX, 011=BX, 100=SP, 101=BP, 110=SI, 111=DI

Source register or memory base

Same codes as Reg field — but when Mod=11, it picks a register. Otherwise picks a memory addressing mode.

Assembler8086.js 

const encodeModRM = (mod, reg, rm) =>
  ((mod & 0b11)   << 6)
  | ((reg & 0b111) << 3)
  | (rm  & 0b111);
PLAIN ENGLISH

Pack three values into one byte using bit shifting.

Put the Mod field (2 bits) into the top two bits of the byte (positions 7-6).

Put the Reg field (3 bits) into the middle (positions 5-3).

Put the RM field (3 bits) into the bottom (positions 2-0). One byte = three fields = all operand info.

Anatomy of the default "Hello 8086" program

The emulator ships with a demo program that writes colored text directly to VGA memory. Let's read it line by line:

DEFAULT_CODE (App.jsx) 

ORG 100h
mov ax, 0xB800
mov es, ax
mov si, hello8086
mov di, 1990
mov cx, 10
print_loop:
  lodsw
  mov [es:di], ax
  add di, 2
  loop print_loop
  hlt
PLAIN ENGLISH

Load this program at address 0x100 in memory (256 bytes in).

Load the number 0xB800 into register AX — the VGA segment number.

Copy AX into ES so ES points to the VGA text screen.

SI points to our "hello8086" data below — the colored character data.

DI=1990: that position on the 80×25 grid is roughly center screen.

CX=10: we will loop exactly 10 times (one per character).

A label marks the top of the loop.

LODSW: load a 16-bit word from [SI] into AX, then advance SI by 2.

Write AX directly into VGA memory at ES:DI — this puts one char+color on screen.

Move DI forward 2 bytes to the next screen cell.

LOOP decrements CX; if CX != 0, jump back to print_loop. Stop when CX hits 0.

HLT — halt the CPU. Program is done.

Character data: encoding ASCII + color in two bytes

Each VGA text cell is exactly 2 bytes. Low byte = ASCII character code. High byte = color attribute (top 4 bits = background color, bottom 4 bits = foreground color):

H
dw 0x0C48
0x48='H' · 0x0C=Light Red
e
dw 0x0E65
0x65='e' · 0x0E=Yellow
l
dw 0x0A6C
0x6C='l' · 0x0A=Lt Green
l
dw 0x0B6C
0x6C='l' · 0x0B=Lt Cyan
o
dw 0x096F
0x6F='o' · 0x09=Lt Blue

Each DW (define word) stores a 16-bit value. Low byte = ASCII code. High byte = DOS color attribute.
