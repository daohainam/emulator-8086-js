The Outside World

BIOS interrupts, VGA memory, keyboard, disk, and the boot sequence

The CPU is not an island

A CPU alone can't do much. It needs to talk to the outside world — the screen, keyboard, disk, speaker. It does this via two channels: memory-mapped I/O and port I/O. Think of it like a hotel: room-to-room calls go through the building wiring (memory-mapped), but calls outside go through the front desk (port I/O).

🖥️

VGA Screen

Write character bytes to 0xB8000 in memory and they appear on screen. No special instruction needed — it's just a memory write.

⌨️

Keyboard

Port 0x60 streams scan codes. The emulator also supports BIOS INT 16h for higher-level keyboard reading.

🔊

PC Speaker

Port 0x61 controls the speaker gate. Writing the frequency divisor to port 0x42 sets the pitch. Implemented via Web Audio API.

💿

Disk

Port 0x70 selects a sector. Port 0x71 transfers it to/from memory at DS:BX. Or use BIOS INT 13h for sector reads/writes.

BIOS interrupts — the original operating system API

Before modern operating systems, programs called the BIOS directly for hardware services. The mechanism is the INT instruction — a software interrupt. Like pressing a service button in an elevator:

INT 10h
Video Services
AH=00 → Set video mode / clear screen
AH=02 → Set cursor position (DH=row, DL=col)
AH=09 → Write character+attr at cursor (AL=char, BL=attr)
AH=0E → Write char in TTY mode (auto-scroll)
INT 16h
Keyboard Services
AH=00 → Wait for a keypress, return in AX
AH=01 → Check if key available (ZF=0 if yes)
AH=02 → Get shift/ctrl/alt key status byte
INT 13h
Disk Services
AH=02 → Read sectors to ES:BX buffer
AH=03 → Write sectors from ES:BX buffer
AH=08 → Get drive geometry (cylinders, heads, sectors)

Inside the BIOS video handler

When your assembly code calls INT 10h, control jumps to handleInt10 in CPU8086.js:

CPU8086.js — handleInt10 (partial) 

const handleInt10 = (e) => {
  const ah =
    (e.reg.AX >> 8) & 0xFF;
  const al =
    e.reg.AX & 0xFF;
  if (ah === 0x0E) {
    const idx = VGA_BASE
      + (e.cursorY * 80
      + e.cursorX) * 2;
    e.mem[idx]     = al;
    e.mem[idx + 1] = 0x07;
    e.cursorX++;
  }
};
PLAIN ENGLISH

The BIOS video interrupt handler — called whenever assembly code runs INT 10h.

Read the high byte of AX (the "function selector")...

...AH tells us which video service was requested.

Read the low byte of AX (the character to print).

AH=0x0E means "TTY write" — print a character at the cursor position.

Calculate the VGA memory address: base 0xB8000 + (row × 80 + column) × 2 bytes per cell.

Write the character byte to that address. Now it shows on screen!

Write 0x07 (white on black) as the color attribute for this cell.

Move the cursor one column to the right — ready for the next character.

The boot sequence — from power-on to running code

When you click the Boot button in the emulator, it replays the exact sequence a real PC does at startup. Understanding this is knowing the blueprint of every operating system ever written:

CPU
CPU (after power-on)

I just turned on. My IP is set to 0xFFF0 — the reset vector. I start executing whatever code is there.

BIOS
BIOS (at reset vector)

I load disk sector 0 (the first 512 bytes of the virtual disk) into memory at address 0x7C00...

BIOS
BIOS

I check the last two bytes of sector 0: are they 0x55 and 0xAA? That is the boot signature — it means "this disk is bootable." If not, I refuse to boot.

CPU
CPU

Boot signature found! BIOS sets CS=0x0000 and IP=0x7C00. I jump to 0x7C00 and start executing the boot sector code. This could be a bootloader, or a tiny OS, or a Hello World.

DISK
Virtual Disk

I am a 128 KB array. Sectors 0-255 each hold 512 bytes. You can write bootable code to me using the Disk Viewer, then click Boot to run it!
