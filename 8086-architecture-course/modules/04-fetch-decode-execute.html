The Heartbeat

Fetch → Decode → Execute: the three-step drum beat every CPU lives by

Every CPU in history runs the same three-step loop

From the 8086 to your MacBook's M-series chip, every CPU runs an infinite loop called the Fetch-Decode-Execute cycle. Think of it like a music box: a tiny pin plucks one note, then the cylinder rotates to the next pin, forever.

📥

FETCH: the first beat

At the start of every instruction, the CPU reads the next byte from memory using CS:IP as the address. IP automatically advances after each read:

CPU8086.js — executeBinaryStep 

const fetch8 = () => {
  const v = readMem8(
    e,
    calcPhys(e.reg.CS, e.reg.IP)
  );
  e.reg.IP =
    (e.reg.IP + 1) & 0xFFFF;
  return v;
};
const fetch16 = () => {
  const lo = fetch8();
  const hi = fetch8();
  return (hi << 8) | lo;
};
PLAIN ENGLISH

fetch8: reads one byte from memory...

...calculate the physical address from segment CS and offset IP...

...read the byte there...

...advance IP by 1 (wrapped to 16 bits so it wraps at 65535)...

...return the byte. IP now points to the byte after the one we just read.

fetch16: reads two bytes to form a 16-bit word...

...read the low byte first (x86 is little-endian)...

...then the high byte...

...combine them: high byte shifted left 8 bits + low byte = 16-bit value.

DECODE: the CPU's giant menu

After fetching the first byte (the opcode), the decoder looks it up in a massive switch statement — one case per possible first byte:

CPU8086.js — executeBinaryStep (simplified) 

let opcode = fetch8();

switch (opcode) {
  case 0xF4:
    // HLT — stop execution
    return false;

  case 0x90:
    // NOP — do nothing
    break;

  case 0xB8: // MOV AX, imm16
    e.reg.AX = fetch16();
    break;
  // ... 200+ more cases ...
}
PLAIN ENGLISH

Fetch the first byte — this is the opcode, the instruction ID number.

A big decision tree: what does this byte mean?

0xF4 is the halt instruction...

...return false to signal "stop running."

0x90 is NOP — No OPeration. Literally do nothing (used for timing padding).

Break to the next iteration of the run loop.

0xB8 is "MOV AX, immediate 16-bit value"...

...fetch the next 2 bytes from memory (the number to load) and put them in AX.

There are 200+ opcodes total. The switch handles all of them.

🔑
The opcode IS the instruction

Machine code is not magic — it is just numbers. 0xF4 always means HLT. 0xB8 always means MOV AX. The CPU's switch statement is its entire vocabulary. That switch statement is what people mean when they say "hardware instruction set."

The run loop: 20 instructions every animation frame

When you click "Run," the emulator uses requestAnimationFrame to run the CPU in bursts — executing about 20 instructions per frame, then handing control back to the browser to update the UI:

App.jsx — runLoop 

const runLoop = useCallback(() => {
  if (!runningRef.current) return;
  for (let i = 0; i < 20; i++) {
    const ok =
      executeStep(eng.current, ctx);
    if (!ok) {
      runningRef.current = false;
      break;
    }
  }
  forceRender();
  requestAnimationFrame(runLoop);
}, []);
PLAIN ENGLISH

The main run loop function — it schedules itself to run every animation frame (~60fps).

If the user clicked Stop, quit immediately.

Execute up to 20 CPU instructions in a tight burst...

...call executeStep() once per instruction. It returns true to continue or false to halt.

If the CPU halted (HLT instruction or error)...

...mark running=false and stop the loop.

After the burst: force the React UI to re-render (show updated registers, VGA screen, etc.)

Schedule the next burst 1/60th of a second later via requestAnimationFrame — keeps the UI responsive.

The Step button — a superpower for understanding

When you click Step in the emulator, it executes exactly one instruction and logs the bytes that were decoded. This is how low-level debugging works:

App.jsx — stepUI 

const stepUI = useCallback(() => {
  const e = eng.current;
  const prevIP = e.reg.IP;
  executeStep(e, ctx);
  const bytesUsed =
    e.reg.IP - prevIP;
  addLog(
    `CS:IP=${toHex(prevIP)}
     | bytes=${bytesUsed}`
  );
  forceRender();
}, []);
PLAIN ENGLISH

The Step function — runs one instruction and logs it.

Get the current CPU state.

Save the current IP address — this is where the instruction started.

Execute exactly one instruction.

Calculate how many bytes the instruction consumed (IP moved forward by this many).

A 1-byte instruction like HLT consumed 1. A MOV AX, imm16 consumed 3.

Log: "CS:IP=0100 | bytes=3" — tells you exactly where you were and how long the instruction was.

Force the UI to refresh so all panels show the updated state.
